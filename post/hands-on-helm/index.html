Hands on Helm

Kubernetes helm
Last updated: Jan 29, 2021

Helm is a package manager for Kubernetes. Just like apt or yum. Helm deploys charts, which we can think of as a packaged application. It is a collection of all your versioned, pre-configured application resources which can be deployed as one unit.

We will be using:

  • minikube (v1.8.2) - We use Minikube to run a single-node Kubernetes cluster inside a Virtual Machine (VM) on local system. Installation
  • kubectl (v1.17.4) - Kubectl is a command line tool for controlling Kubernetes clusters. Installation
  • helm (v3.2.0) - Installation
  • socat - Socat cat (apt install socat)

Lets Start

helm create demo

This will create a folder in your current working directory. That folder will contains all the configuration of your application. Basically most of the files in your chart directory will be .yaml files.

Above is just snap of my chart directory. Here,

  • Chart.yaml - Contains the chart version and application version.
  • values.yaml - Contains all the values to be passed throughout the application.
  • .helmignore - It is just like .dockerignore & .gitignore.
  • /templates - When Helm renders the charts, every file in this folder is passed through the template engine.

You can see templates directory contains templates about hpa, services and deployment.

In above snap all the values will be taken from values.yaml file.

lets deploy our first application.

**In values.yaml file we are using nginx image, which comes with boilerplate.

Start cluster

minikube start

First we will lint our charts.

helm lint ./demo

Following command will console all the yaml files after template engine.

helm install --dry-run --debug --generate-name ./demo

**generate-name will give a random name to application.

Deploy the app

helm install test ./demo

Verify deployment

kubectl get all

At last

kubectl --namespace default port-forward $POD_NAME 8080:80

Go to 127.0.0.1:8080
